COE892 - Project Proposal - Group 14
Problem:
Finding parking is often inefficient and frustrating, especially in busy areas such as campuses, shopping centers, downtown locations, event venues, etc. Drivers usually have no way of knowing which parking spots are available, so they end up driving around searching for a space. This increases traffic, wastes time, causes unnecessary stress, and in some cases, it can even lead to conflicts between drivers competing for the same spot.


From a driver’s point of view, the lack of real-time information and reservation options makes trip planning difficult. From a business or parking operator’s perspective, parking spaces are not always used efficiently. For example, during peak hours, demand is high and unmanaged, while during off-peak hours, many spaces remain unused. Without demand-based pricing, operators may lose revenue and have limited insight into parking patterns. These issues show the need for a Smart Parking System that can track availability in real time, allow reservations, and adjust pricing based on demand and peak hours.


Solution:
We will develop a distributed software project to allow drivers to see available parking spots and make reservations. Our proposed solution will be a microservice architecture, including a FastAPI component as an endpoint for end-users and RabbitMQ to receive sensor events. 
The components of the project will be:
* Simulated Sensors - The simulated sensors will send parking spot events (NowOccupied, NowAvailable) to the message queue.
* MessageQueue - The message queue will receive events from the simulated sensors. These events will be consumed by the inventory service.
* InventoryService - This service will hold and organize the state of all parking spots and reservations. It will consume events from the message queue and update the parking spot states accordingly. It will also have methods to expose this state to the FastAPI component, so that users can see which spots are available. 
* PricingService - This service will use availability, reservation and time data to create dynamic pricing for parking spots.
* FastAPI - This will be the API endpoint between the users and the backend. It will receive requests from users on the UI and retrieve availability and prices, as well as make reservation requests.
* Mobile App - This will be a basic UI for drivers to see parking spot availability and pricing, and to make reservations.


The course technologies and techniques will be used in the following ways:
* RabbitMQ will be used as the message queue technology
* FastAPI will be used as the API endpoint
* Each service and component can be containerized and deployed separately on the cloud
* FastAPI, pricing service and inventory service will have gRPC calls to connect them
* There will be concurrency checks to ensure that parking spots can’t be double booked
